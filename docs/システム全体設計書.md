# 医療機器薬事承認支援アプリ システム全体設計書

- 版数: 1.0
- 作成日: 2026-07-03
- フェーズ: 設計（実装は次フェーズ）

---

## ① システム全体アーキテクチャ

### 方針

- モノレポ + Clean Architecture。フロント/バック/共有パッケージを分離し、保守性・拡張性を最優先。
- インフラは無料枠で開始し、有料化はスケール設定変更のみで対応（構成変更不要）。

### 構成図（論理）

```
[PC / iOS / Android ブラウザ (PWA)]
        │ HTTPS
        ▼
[Vercel] Next.js (App Router)
  ├ SSR/ISR ページ・静的アセット・PWA Service Worker
  └ BFF (Next.js Route Handlers) ── 認証セッション管理
        │ REST (JSON) / OpenAPI
        ▼
[Railway]
  ├ API Server: Node.js (NestJS)
  ├ Worker: 法規制情報クローラ・埋め込み生成（cron / queue）
  ├ PostgreSQL 16（+ pg_trgm, pgvector 拡張）
  └ Redis（キャッシュ・ジョブキュー BullMQ）
        │
        ▼
[外部] PMDA / FDA / EU 等の公開情報源、LLM API（AI機能）
```

### 技術選定

| 層       | 技術                                           | 理由                                                      |
| -------- | ---------------------------------------------- | --------------------------------------------------------- |
| フロント | Next.js 15 / TypeScript strict / Tailwind CSS  | Vercel最適、PWA/レスポンシブ容易                          |
| バック   | NestJS / TypeScript strict                     | DI・モジュール分割でClean Architectureと親和              |
| DB       | PostgreSQL (Railway)                           | 全文検索(pg_trgm)+ベクトル(pgvector)を1DBで完結し無料枠内 |
| キュー   | Redis + BullMQ                                 | クローラ・AI処理の非同期化                                |
| CI/CD    | GitHub Actions → Vercel / Railway 自動デプロイ | main=本番、develop=プレビュー                             |

### 無料運営の担保

- Vercel Hobby + Railway 無料枠 + 単一Postgres/Redis。Elasticsearch等の専用検索基盤は採用しない（pg内で代替）。
- 有料プラン追加時は Stripe を後付け可能な `plan` カラム・エンタイトルメント層のみ先行設計（⑦参照）。

## ② フロントエンド構成

### アーキテクチャ

- Next.js App Router。Server Components を既定、対話部のみ Client Components。
- 状態管理: サーバ状態=TanStack Query / UI状態=Zustand（最小限）。
- フォーム: React Hook Form + Zod（スキーマは共有パッケージ `packages/shared` からimport）。

### レイヤ構成（Feature-Sliced）

```
src/
  app/          # ルーティング・レイアウト
  features/     # 機能単位（search, learning, regulation, qms, ai-chat ...）
    <feature>/
      api/        # APIクライアント（自動生成型使用）
      components/
      hooks/
      model/      # 型・ロジック
  shared/       # UIキット・ユーティリティ・デザイントークン
  entities/     # ドメインモデル表示部品（Device, Regulation, JMDN ...）
```

### PWA

- `next-pwa` 相当の Service Worker: アプリシェル+閲覧済み法令のオフラインキャッシュ（Stale-While-Revalidate）。
- Web App Manifest（アイコン、standalone、テーマカラー、日英名称）。
- iOS/Android はホーム画面追加で対応（ストア配布は将来課題、本設計では対象外）。

### レスポンシブ

- ブレークポイント: sm 640 / md 768 / lg 1024 / xl 1280。
- モバイル: 下部タブバー、PC: 左サイドバー（同一ルーティング、レイアウトのみ切替）。

## ③ バックエンド構成

### NestJS モジュール構成（Clean Architecture）

```
src/
  modules/
    auth/           # 認証・セッション
    users/          # ユーザー・組織・ロール
    regulations/    # 法令・通知・ガイダンス管理
    jmdn/           # 一般的名称・JMDN
    search/         # 検索API（⑩）
    learning/       # 学習コンテンツ・進捗
    workflow/       # 実務支援（申請チェックリスト等）
    ai/             # AIチャット・RAG
    ingestion/      # 薬事情報更新パイプライン（⑨）
    notifications/  # 更新通知
  core/
    domain/         # エンティティ・値オブジェクト（フレームワーク非依存）
    usecases/       # アプリケーションサービス
  infrastructure/
    database/       # Prisma / リポジトリ実装
    external/       # 外部API・LLMクライアント
  common/           # guard, interceptor, filter, pipe
```

- 依存方向: infrastructure → usecases → domain（逆流禁止）。リポジトリはインターフェースを domain 側に定義。
- ORM: Prisma（マイグレーション管理・型安全）。
- Worker はAPIと同一コードベース別プロセス（Railway 2サービス目、無料枠内は同居可）。

## ④ データベース構成

### 主要テーブル（PostgreSQL）

**ユーザー系**

- `users` (id, email, name, locale, plan, created_at …)
- `organizations` (id, name, type[maker/marketing_holder/importer/academic…])
- `memberships` (user_id, org_id, role)

**法規制系（多国対応の中核）**

- `jurisdictions` (id, code[JP/US/EU/UK/CA/AU/CN/KR/TW/SG], name, authority[PMDA/FDA/…])
- `regulations` (id, jurisdiction_id, type[law/ordinance/notice/guidance/standard], title, doc_number, status[active/amended/repealed], effective_date, source_url)
- `regulation_versions` (id, regulation_id, version_no, published_at, effective_from, effective_to, full_text, summary, change_summary) — **法改正はバージョン行追加で表現し、履歴を不変保持**
- `regulation_sections` (id, version_id, path[条・項・号の階層], heading, body, embedding vector) — RAG・全文検索の単位
- `regulation_relations` (from_id, to_id, relation[amends/implements/refers/supersedes])

**分類系**

- `device_classifications` (id, jurisdiction_id, scheme[JMDN/FDA_ProductCode/EMDN/GMDN…], code, name, class[I–IV等], definition) — 国別スキームを同一テーブルで抽象化
- `classification_mappings` (from_id, to_id, confidence) — JMDN↔FDA等の相互参照

**学習系**

- `courses` / `lessons` / `quizzes` / `quiz_questions`
- `user_progress` (user_id, lesson_id, status, score, completed_at)

**実務支援系**

- `checklists` (id, jurisdiction_id, procedure[届出/認証/承認/510k/CE…], items jsonb)
- `user_projects` (id, org_id, name, device_class, target_jurisdictions[])
- `project_tasks` (project_id, checklist_item_ref, status, due_date, assignee)

**共通**

- `tags` / `taggings` (polymorphic: regulation, lesson, classification)
- `audit_logs` (actor, action, target, at)
- `ingestion_jobs` (source, status, diff_summary, run_at) — ⑨の実行履歴

### 設計原則

- 全テーブル `created_at/updated_at`、論理削除は法規制系のみ（status で表現、物理削除しない）。
- 多言語: `title_i18n jsonb` 方式ではなく原文＋`translations` テーブル（将来の翻訳品質管理のため）。
- ID: UUIDv7。

## ⑤ API構成

### 方針

- REST + OpenAPI 3.1（NestJS から自動生成→フロントの型を codegen）。バージョンは URL プレフィックス `/api/v1`。

### 主要エンドポイント

```
POST   /api/v1/auth/{signup|login|logout|refresh}
GET    /api/v1/me

GET    /api/v1/jurisdictions
GET    /api/v1/regulations?jurisdiction=&type=&q=&tag=
GET    /api/v1/regulations/:id            # 最新版
GET    /api/v1/regulations/:id/versions   # 改正履歴
GET    /api/v1/regulations/:id/diff?from=&to=

GET    /api/v1/classifications?scheme=JMDN&q=
GET    /api/v1/classifications/:id/mappings

GET    /api/v1/search?q=&scope=all|regulation|jmdn|generic-name|learning&tag=
GET    /api/v1/search/suggest?q=

GET    /api/v1/courses / lessons / quizzes（学習系CRUD＋進捗POST）

GET/POST /api/v1/projects, /projects/:id/tasks   # 実務支援

POST   /api/v1/ai/chat            # SSEストリーミング、RAG回答＋出典
POST   /api/v1/ai/classify        # 機器概要→候補分類提示

GET    /api/v1/updates?since=     # 薬事情報更新フィード
POST   /api/v1/subscriptions      # 更新通知購読
```

- 共通仕様: カーソルページネーション、`ETag` キャッシュ、RFC 9457 Problem Details エラー、レートリミット（Redis）。

## ⑥ AI機能構成

### 機能

1. **薬事QAチャット**: RAG。回答には必ず出典（regulation_sections への参照＋版・施行日）を付与。
2. **分類支援**: 機器説明→JMDN/クラス候補提示（検索+LLM再ランク）。
3. **改正差分要約**: 新版取込時に Worker が change_summary を生成（⑨と連携）。
4. **学習支援**: レッスン内容に基づく解説・クイズ生成（人手レビュー後公開）。

### RAG パイプライン

```
質問 → クエリ拡張（同義語: 医療機器/medical device 等）
     → ハイブリッド検索（pgvector cos類似 + pg全文検索、RRF融合）
     → 上位k=8 section → LLMへコンテキスト注入 → 出典付き回答（SSE）
```

- 埋め込み: 多言語対応モデル（例: text-embedding系）。生成: LLM APIをプロバイダ抽象化層（`infrastructure/external/llm`）経由で利用し、モデル差替可能。
- ガードレール: 「本回答は参考情報であり規制上の判断はPMDA等へ確認」の定型免責、対象国・版の明示、根拠なし回答の禁止（検索ヒット0件時は回答拒否テンプレート）。
- コスト制御: 回答キャッシュ（質問正規化ハッシュ）、無料プランは日次回数制限（エンタイトルメントで制御）。

## ⑦ 認証・権限管理構成

- 認証: Auth.js（メール+パスワード、Google/Microsoft OAuth）。セッションは JWT（短命 access 15分 + refresh 30日、httpOnly cookie）。
- 認可: RBAC。
  - システムロール: `admin`（運営）/ `editor`（法規制データ編集）/ `user`
  - 組織ロール: `org_admin` / `member`（membershipsで保持）
  - 職能属性: regulatory / qa / safety / sales / design / medical / academic — 権限ではなく**表示パーソナライズ**に使用（権限モデルを汚さない）
- エンタイトルメント層: `plan`（free/pro/enterprise）→機能フラグ（AI回数、プロジェクト数、通知数）。課金実装前でも判定層のみ存在させ、将来 Stripe webhook で plan を更新するだけにする。
- API保護: NestJS Guard（JWT検証→RBAC→エンタイトルメント）の3段。

## ⑧ 法規制データ管理構成

- **正: regulation_versions**（④）。原文・版・施行日・出典URLを不変保持し、編集は新版追加のみ。
- 編集ワークフロー: `draft → review → published`。editor が取込結果を校閲して公開（AI要約も同フロー）。
- 構造化: 法令は条文階層（regulation_sections.path 例 `第23条の2/第1項/第2号`）、通知・ガイダンスは見出し単位で分割。
- 関係グラフ: regulation_relations で「政令←法律」「通知→改正対象」「基準↔ISO/IEC」を表現し、画面で関連表示・AIで根拠遡及に利用。
- 各国拡張: jurisdiction_id で完全分離。国別の文書タイプ差（例: FDA Guidance, EU MDCG）は `type` + `subtype` で吸収し、スキーマ変更不要。

## ⑨ 国内外の薬事情報更新アーキテクチャ

```
[cron (Railway Worker, 日次)]
  → Source Adapter（国別プラグイン: PMDA通知一覧, FDA Federal Register/510k DB,
     EUR-Lex/MDCG, MHRA, Health Canada, TGA, NMPA, MFDS, TFDA, HSA）
  → 取得（HTML/PDF/RSS/API）→ 正規化（共通スキーマへ変換）
  → 差分検出（doc_number+ハッシュ比較）
  → 新規/改正なら regulation_versions に draft 登録
  → AI差分要約生成 → editor レビューキューへ
  → published 時: 購読ユーザーへ通知（アプリ内+メール）、埋め込み再生成
```

- Adapter はインターフェース `RegulatorySource { fetchList(); fetchDocument(); normalize(); }` を実装するプラグイン方式。国追加=Adapter追加のみ。
- 失敗時リトライ・`ingestion_jobs` に監査記録。ソースのHTML構造変化はジョブ失敗アラートで検知。
- MVP は JP のみ稼働、他国は Adapter 空実装（スキーマ・UIは最初から多国前提）。

## ⑩ 検索基盤

### 単一Postgresでのハイブリッド検索

| 検索           | 実装                                                                             |
| -------------- | -------------------------------------------------------------------------------- |
| 全文検索       | `tsvector`（日本語は pg_bigm/pg_trgm によるN-gram）＋見出し重み付け              |
| 法令検索       | regulations/sections 対象。ファセット: 国・文書タイプ・施行日・状態              |
| 一般的名称検索 | device_classifications (scheme=JMDN) の name/definition。前方一致+あいまい(trgm) |
| JMDNコード検索 | code 完全一致/前方一致                                                           |
| タグ検索       | taggings JOIN。複数タグAND/OR                                                    |
| 意味検索(AI)   | pgvector（⑥と共用）                                                              |

- 統合検索API `/search?scope=` が上記を単一UIに束ね、スコアはRRFで融合。
- サジェスト: 検索ログ+名称辞書から trgm 類似。
- インデックス: GIN(tsvector), GIN(trgm), HNSW(vector)。データ量（法令数万件規模）ではpgで十分、将来必要時のみ検索エンジンを差替可能なよう `search` モジュールにリポジトリ抽象を置く。

## ⑭ デザインシステム

- 原則: 余白主導・階層はタイポグラフィで表現・装飾最小・1画面1主操作（主要操作は常に1つ）。
- トークン（packages/ui、CSS変数でライト/ダーク自動切替）:
  - 色: `--bg / --surface / --text / --text-secondary / --accent(青系1色) / --danger / --border`。彩色はアクセント1色のみ。
  - タイポ: システムフォント（SF Pro/Noto Sans JP相当）。サイズ階層 12/14/16/20/24/32、行間1.6。
  - 角丸 8/12、影は2段階のみ、間隔は4pxグリッド。
  - モーション: 150–250ms ease-out、意味のある遷移のみ。
- コンポーネント: Button, Input, SearchBar, Card, List, Tabs, Badge(クラス/国表示), Modal, Toast, Skeleton, EmptyState, DiffViewer(条文差分), CitationChip(AI出典)。
- アクセシビリティ: WCAG 2.2 AA、コントラスト4.5:1、フォーカスリング、44pxタップ領域。
- ダークモード: OS追従+手動切替（S19）。
