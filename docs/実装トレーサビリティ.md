## 位置づけ

本書は設計書そのものを置き換えるものではない。設計書は引き続き唯一の仕様書であり、本書は「設計書の各項目が実装のどこで・どのように実現されているか」を追跡できるようにするための対応表である。実装がやむを得ず設計書から逸脱している箇所は、逸脱の理由・承認状況とあわせて明記する（本プロジェクトの運用ルール「設計変更は理由を説明し承認されるまで実装を変更しない」に基づき、これまで全て事前にユーザー承認を得たうえでの逸脱であることを確認済み）。

検証方法: GitHub REST API（`api.github.com`、`raw.githubusercontent.com`）による`main`/`develop`ブランチの実ファイル直接確認、および過去セッションの検証済みメモリ記録（PR番号・コミットSHA・検証コマンド結果を伴うもの）の突合による。ローカルOneDriveフォルダの`.git`は既知の破損状態（[[feedback_fuse_mount_file_writes]]）のため検証には使用していない。Phase7部分は個々のPRマージ・CI green・本番デプロイ確認をメモリ記録（[[project_phase7_saas_design_change]]、[[project_phase7_2_lifecycle_master_data_status]]）で突合したが、GitHub API側のキャッシュ不整合（同一エンドポイントが数分内に異なる古い応答を返す事象を今回複数回観測）があり、コミット単位の網羅的な再検証は今回実施していない。

## 7. ユーザー承認済み設計逸脱一覧（まとめ）

以下はいずれも実装時にユーザーへ確認・承認を得たうえでの設計書からの逸脱であり、無断の仕様追加ではない。

1. `checklists`マスタテーブルを作らず、`project_tasks`を自由入力のタスク管理とした（AIによる規制内容の推測実装リスク回避）。
2. `update_subscriptions`/`notifications`/`ai_chat_sessions`/`ai_chat_messages`/`tags`/`taggings`を設計書④に無い形で追加。
3. `tags`/`taggings`のpolymorphic対象は`LESSON`のみ実装（regulation/classificationは編集画面未実装のため保留）。
4. 改正差分要約(`diffSummary`/`changeSummary`)はAI保留期間中のため非AIの簡易自動要約で代替。
5. 認証アーキテクチャはNestJSを正としAuth.jsをBFFセッション管理層のみに限定。next-authはv5 beta採用。
6. refresh token失効はRedis失効リスト方式（DBテーブル追加なし）。
7. PWAはnext-pwaではなくServwist採用（next-pwaが開発終了のため）。
8. `user_projects.organizationId`をnullable化し`userId`所有者を追加（signup時の組織自動作成が無いため）。
9. 多国Adapterを設計書の「MVP=JPのみ」からPhase6でUS(FDA)/EU(MDCG)まで拡張。
10. published時のメール通知は保留、アプリ内通知のみ実装。
11. エンタイトルメント上限値(FREE 3/PRO 20/ENTERPRISE無制限)はユーザー承認により決定（設計書はカテゴリのみ規定）。
12. BUSINESS上限値はPROと同値(20件)とユーザー承認により決定（設計変更書にBUSINESS値の明記が無かったため）。
13. admin向けcomplimentary手動付与は「組織へ付与」ではなく「対象ユーザーへ付与し、organizationIdは記録用タグ」という実装（エンタイトルメント判定がSubscriptionテーブルではなく`users.plan`直接参照のため）。失効時は一律`users.plan`をFREEへ戻す（STRIPE契約との併用ケースは未対応、技術的負債としてコード内明記）。
14. LifecycleTemplateのpublishはregulation_versionsと異なり「1エンティティ不変・単純status flip」（旧版クローズ処理なし）。PUBLISHED後は編集・削除とも不可。
15. 工程マスタのadmin CRUDには監査ログ(RecordAuditLogUsecase)を追加していない（course/tag/subscription等の既存admin CRUDの多数派パターンに合わせた。全admin操作への監査ログ拡大は別途要ユーザー判断）。
16. 初期マスタデータのうち18工程のうちREG_STRATEGY〜REIMBURSEMENTの5工程のみ期間/費用を記載し、残り13工程はduration/cost系を全てnull（根拠のない目安値を書くよりnullが「推測ではなく根拠を提示」原則に合致すると判断）。
